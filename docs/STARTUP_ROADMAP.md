# AgentTrust Startup Roadmap

## Core Reframe

AgentTrust should not be sold as a generic identity layer for all agents. The stronger framing is:

"AgentTrust is the trust and policy infrastructure that decides which agents get access, visibility, and money."

That turns the product from a dashboard into operational infrastructure for platforms that host or route autonomous agents.

## Initial Wedge

The first buyer is not the agent itself. The first buyer is the operator that must decide:

- which agents get listed
- which agents get routed to valuable work
- which agents can move from sandbox to production
- which agents receive instant payout versus holdback

The best early customers are:

1. Agent marketplaces
2. Agent execution platforms
3. Enterprise AI gateways

## Why They Buy

These operators need a shared decision layer for:

- identity continuity
- provenance visibility
- release risk
- reputation weighting
- scoped machine access
- payout and settlement eligibility

Without that layer they rely on ad hoc allowlists, manual review, and weak self-reported metadata.

## Product Surface That Matters

To become a business, AgentTrust needs to influence production decisions rather than only display trust metadata.

### Required decision surfaces

- listing and ranking
- routing preference
- API rate limits
- high-risk action approvals
- payout holdbacks
- settlement eligibility
- escalation and dispute review

### Required buyer-facing outputs

- policy check API
- trust/risk score
- release warnings
- provenance status
- issuer-weighted attestation score
- partner-verification signals

## Monetization Path

### 1. Policy API

Platforms pay for runtime policy decisions:

- can this agent access the workflow?
- should this agent be routed to this task?
- should this payout be held?

### 2. Enterprise Trust Compliance

Enterprises pay for:

- provenance exports
- partner-specific policy controls
- audit trails
- incident and dispute visibility

### 3. Economic Security Controls

Networks pay for:

- bonded actions
- settlement gates
- payout holdbacks
- dispute and slashing workflows

## Product Phases

### Phase 1: Credible prototype

- self-registration
- passports
- attestations
- release manifests
- network policy API

### Phase 2: Sellable B2B trust layer

- partner-scoped policies
- weighted attestations
- marketplace routing API
- release risk and payout warnings
- dispute workflows

### Phase 3: Network moat

- partner integrations across multiple ecosystems
- portability of trust across platforms
- proprietary trust graph
- economic security primitives
- historical reputation and release intelligence

## Key Discipline

Do not optimize for "more agents registered." Optimize for:

- higher quality routing decisions
- fewer bad executions
- lower fraud or abuse
- more confidence in payout and settlement
- lower manual review burden for operators
